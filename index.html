📖 System Architecture & Theoretical Reasoning

1. Rate Limiter — Redis (Key-Value Store)

Working: Every incoming HTTP request is identified by IP address. Redis stores a key rate_limit:<ip> with an integer counter. The INCR command atomically increments this counter. On first call, an EXPIRE of 60s is set. When the counter exceeds 5, a 429 Too Many Requests response is returned. After 60s, the key vanishes automatically.


Structural Concept: Redis is a single-threaded, in-memory key-value store. Its data model is: STRING key → INTEGER value. No tables, no joins, no schema. Pure atomic operations on named counters — exactly what rate limiting needs.


Reasoning: SQL would require a table with IP, count, window_start columns, a transaction with SELECT FOR UPDATE, and a cron to clean expired rows. Redis replaces all of this with two commands: INCR and EXPIRE. The sub-millisecond response ensures rate checking adds near-zero latency to every request.

2. Inventory Management — MongoDB (Document Store)

Working: Each product is stored as a JSON-like BSON document with a unique _id. Documents can have different shapes — a laptop document has specs.cpu while a chair document has specs.material. The API performs CRUD using MongoDB's find, insertOne, updateOne, and deleteOne operations.


Structural Concept: MongoDB organises data as a hierarchy: Database → Collection → Document. A document is a BSON object with no enforced schema. Fields can be nested, arrays can be embedded. This is an unstructured/semi-structured model.


Reasoning: Inventory is naturally heterogeneous. A relational DB forces a Products table with 50 nullable columns (most NULL for any given row) or complex entity–attribute–value patterns. MongoDB stores each product's real attributes only. Rich queries still work: find all Electronics under ₹1000 in one query.

3. Product-Manufacturer Correlation — Neo4j (Graph DB)

Working: Entities (Product, Manufacturer, Category, Region) are nodes. Their connections (MANUFACTURED_BY, SUPPLIES, BELONGS_TO, LOCATED_IN) are directed edges with properties. Neo4j's Cypher query language allows pattern matching: MATCH (p:Product)-[:MANUFACTURED_BY]->(m) traverses relationships without JOINs.


Structural Concept: A property graph consists of: nodes (vertex with label + properties) and relationships (directed edge with type + properties). Neo4j stores adjacency lists natively — each node contains direct pointers to its relationships, making traversal O(relationship count), not O(table size).


Reasoning: "Which products share a component supplier with a competitor?" requires a 3-hop traversal. In SQL this is 3 JOINs across 4 tables with exponential row growth. In Neo4j it's a single MATCH pattern. Supply chain analysis, manufacturer correlation, and shortest-path queries are native graph problems that graph DBs solve in milliseconds.
